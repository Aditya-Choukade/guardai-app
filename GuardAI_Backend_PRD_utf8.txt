1. Product overview
GuardAI is a stateless REST API that classifies user-submitted text messages and URLs as Safe, Suspicious, or Dangerous using Claude Sonnet. Every response includes a plain-English explanation of the threat, structured flag tags, and an actionable next step for the user.
The backend is the single source of truth for all intelligence. The React frontend, and any future browser extension, are thin clients that call one endpoint — /analyze — and render the response.
NOTE
This PRD covers the English-only MVP. Multilingual support (Hindi, Hinglish, regional languages) is a post-hackathon addition and is explicitly out of scope here.
1.1 Design principles
Stateless by default — no user accounts, no scan history, no database required for core functionality
Single endpoint philosophy — /analyze handles everything; internal complexity is hidden from the client
Claude sees the full picture — message text and URL scan results are sent to Claude together in one prompt, not in separate calls
Fail gracefully — if VirusTotal is rate-limited or unavailable, Claude still returns a verdict based on message content alone
Demo reliability first — every edge case must degrade cleanly, never throw a 500 during a live demo
1.2 North star
GOAL
A judge pastes a scam SMS into the demo UI, hits Analyze, and within 5 seconds sees a Dangerous verdict with a specific, accurate English explanation — without any manual configuration.
2. System architecture
2.1 Component overview
The backend consists of three logical layers that run in sequence on every /analyze request:
Layer
Component
Responsibility
Ingestion
FastAPI endpoint
Validates input, extracts URLs, dispatches parallel sub-tasks
Intelligence
VirusTotal + WHOIS
URL reputation, domain age check, known-bad hash lookup
Intelligence
Claude Sonnet
Reads message + URL results, returns structured JSON verdict
Response
Response builder
Merges all outputs into unified API response shape
2.2 Request processing order
When a POST /analyze request arrives, the backend executes this sequence:
Step 1 — Input validation: reject empty strings, strip whitespace, enforce 4000-char max
Step 2 — URL extraction: regex scan of text for all http/https URLs
Step 3 — Parallel intelligence: fire VirusTotal + WHOIS calls for each URL concurrently (asyncio.gather)
Step 4 — Unified Claude call: send message text + all URL scan results in one structured prompt
Step 5 — Response assembly: merge Claude output with raw URL data into final response shape
Step 6 — Return: respond within 5-second SLA; log processing_ms
WHY
Sending everything to Claude in one call produces better verdicts than two separate calls. Claude can reason that a message claiming urgency plus a 3-day-old domain is Dangerous, when each signal alone might only be Suspicious.

3.5 Threat flag taxonomy
Claude returns flags from this fixed set. The frontend renders each flag as a chip on the verdict card. Restricting to a fixed set prevents Claude from inventing inconsistent labels.

Meaning

Message asks for an OTP, PIN, or password

Creates artificial time pressure ('act now', 'expires today')

Impersonates a bank, government body, or known brand

URL flagged by VirusTotal or has anomalous characteristics
newly_registered_domain
Domain registered within the last 30 days
prize_scam
Claims user has won a lottery, prize draw, or cashback
kyc_fraud
Falsely claims KYC verification is required or will expire
impersonation
Pretends to be a person of authority (bank manager, government officer)
malware_link
URL confirmed malicious by 5+ VirusTotal engines
4. Claude prompt architecture
The quality of every verdict depends entirely on this prompt. Spend the first 90 minutes of the build session getting this right with the test corpus before writing any other code.
4.1 System prompt
  
You are a cyber-fraud detection expert specializing in Indian
digital scam patterns: UPI fraud, fake KYC, phishing via
WhatsApp/SMS, prize scams, and bank impersonation.
You will receive a JSON object with:
  - "message": the text to analyze
  - "url_scan_results": array of VirusTotal/WHOIS results
                        for any URLs found in the message
Return ONLY a valid JSON object with this exact structure:
{
  "verdict": "Safe" | "Suspicious" | "Dangerous",
  "confidence": <float 0.0-1.0>,
  "explanation": "<2 sentences max. Cite the specific red flag.",
  "flags": [<flag strings from allowed list only>],
  "action": "<what the user should do next>"
}
Allowed flags: otp_request, urgency_pressure, fake_sender,
suspicious_url, newly_registered_domain, prize_scam,
kyc_fraud, impersonation, malware_link
Rules:
- Dangerous = clear fraud intent OR confirmed malicious URL
- Suspicious = partial signals, not conclusive
- Safe = no threat signals; legitimate communication pattern
- Legitimate OTP from a known bank format → lean Safe
- Never return markdown. Return only valid JSON. No preamble.
4.2 User message shape
  
{
  "message": "Congratulations! You won Rs 50,000. Click...",

    {

5. Error handling
All errors follow RFC 7807 Problem Details. Never return a plain string error — always return a JSON object so the frontend can handle it consistently.
HTTP code
Error type
When it fires
400
invalid_input
Empty text, text > 4000 chars, or missing required field
422
validation_error
Pydantic schema violation (wrong type, missing field)
429
rate_limited
VirusTotal free tier rate limit hit — Claude-only verdict returned
503
claude_unavailable
Claude API unreachable — retry after 2s with exponential backoff
500
internal_error
Unexpected exception — returns safe fallback, logs full trace
CRITICAL
On VirusTotal 429 rate limit, do NOT return an error to the client. Silently fall back to Claude-only verdict and set virustotal_score to null. The demo must never fail because of a rate limit.
Standard error response shape
  
{
  "type": "invalid_input",
  "title": "Text field is required",
  "status": 400,
  "detail": "The text field must be a non-empty string."
}
6. Performance requirements
P95 response time
< 5 seconds end-to-end (including Claude call)
URL scans
VirusTotal + WHOIS run in parallel via asyncio.gather — not sequentially
Claude timeout
8 second hard timeout; if exceeded, return Suspicious verdict with explanation
In-memory URL cache
Cache VirusTotal results by URL hash for 15 minutes — prevents rate limits during demo
Max input length
4000 characters; reject with 400 above this
Concurrent requests
10 simultaneous /analyze calls on Railway free tier

9. Risk register
HIGH
Claude returns inconsistent verdicts on edge cases
Especially OTP messages from legitimate banks may be incorrectly flagged as Dangerous.
Mitigation: Run the full test corpus before writing any UI code. If accuracy < 85%, refine the system prompt. Do not proceed to P2 until P1 passes.
HIGH
VirusTotal rate limit during live demo
Free tier allows 4 requests/minute. A judge testing multiple URLs in quick succession will hit the limit.
Mitigation: Implement 15-minute in-memory cache keyed by URL. On 429, silently return Claude-only verdict. Never surface rate limit errors to the user.
MED
Claude API response is not valid JSON
Rarely, Claude may include preamble text or markdown fences around the JSON block.
Mitigation: Wrap JSON parse in try/except. Strip ```json fences before parsing. If parse fails after 2 retries, return a Suspicious verdict with a generic explanation.
MED
WHOIS lookup fails or times out
Some TLDs do not have consistent WHOIS coverage. Timeouts can add 2-3 seconds to response time.
Mitigation: Set a 2-second hard timeout on all WHOIS calls. Treat a timeout as domain_age_days: null — do not fail the overall request.
LOW
Slow cold start on Railway/Render free tier
Free tier instances spin down after 15 minutes of inactivity. First request after sleep can take 10-30 seconds.
Mitigation: Keep the /health endpoint warm with a simple cron ping. Alternatively, wake the instance before the demo starts.
